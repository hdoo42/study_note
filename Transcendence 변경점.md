### 1. 프로젝트 개념 및 목표 변화 (Subject & Description)
기존의 정해진 스펙을 구현하는 것에서 벗어나, 창의성과 팀워크를 강조하는 방향으로 전면 개편되었습니다.

* **자유 주제 (Creative Freedom):** 더 이상 Pong 게임만 만들 필요가 없습니다. 체스, 소셜 네트워크, 협업 도구 등 **웹 기술을 활용한 어떤 어플리케이션**이든 만들 수 있습니다. (물론 Pong을 선택하여 구현하는 것도 여전히 가능합니다.)
* **팀 역할 분담 의무화:** 팀원(4~5명)은 반드시 아래 역할을 분담하고 이를 문서화해야 합니다.
    * **Product Owner (PO):** 제품 비전 정의, 기능 우선순위 결정.
    * **Project Manager (PM):** 일정 관리, 팀 조율.
    * **Technical Lead:** 기술 아키텍처 결정, 코드 품질 관리.
    * **Developers:** 기능 구현.
* **목표 키워드 변경:** Specs(명세) → Group(그룹), Creativity(창의성), Framework(프레임워크).

### 2. 평가 시스템 및 모듈 (Scale & Modules)
점수 산정 방식과 모듈 카테고리가 완전히 바뀌었습니다.

* **점수제 도입:** 기존의 필수 모듈 리스트 대신, **총 14점**을 채우는 방식으로 변경되었습니다.
    * **Major Module:** 2점 (난이도 높음/주요 기능)
    * **Minor Module:** 1점 (부가 기능)
    * 14점을 초과하면 보너스 점수로 인정됩니다.
* **새로운 모듈 카테고리:**
    * **Web:** 프레임워크 사용(Front/Back), 실시간 기능, API, ORM 등.
    * **Accessibility:** 접근성(WCAG), 다국어 지원, RTL 지원.
    * **User Management:** OAuth, 2FA, 조직/권한 관리.
    * **AI:** AI 상대, RAG, LLM, 추천 시스템.
    * **Cybersecurity:** WAF/ModSecurity, Vault.
    * **Gaming:** 웹 게임 구현, 원격 플레이, 3인 이상 멀티플레이, 3D 그래픽.
    * **Devops:** ELK 스택, Prometheus/Grafana, 마이크로서비스.
    * **Blockchain:** 토너먼트 점수 기록 (Solidity/Avalanche 등).
* **평가 방식 강화:**
    * **팀 전원 참석 필수:** 평가 시 팀원이 불참하면 1인당 -10점 페널티.
    * **역할 인터뷰:** 각 팀원이 자신의 역할(Role)과 기여도를 명확히 설명해야 함.
    * **기여도 확인:** Git 커밋 내역을 통해 실제 협업 여부 확인.

### 3. 기술적 필수 요구사항 (Mandatory Part)
프로젝트 주제는 자유지만, 반드시 지켜야 할 기술적 제약사항이 재정의되었습니다.

* **필수 구성:** 프론트엔드 + 백엔드 + 데이터베이스.
* **CSS 프레임워크:** Tailwind, Bootstrap 등 최신 CSS 솔루션 사용 필수.
* **보안:**
    * 모든 민감 정보(API Key 등)는 `.env` 파일로 관리 (Git 업로드 금지).
    * 백엔드는 HTTPS 사용 필수.
    * 비밀번호 해싱 및 입력값 검증 필수.
* **배포:** Docker/Podman 등을 이용한 **컨테이너화** 및 단일 명령어 실행.
* **브라우저:** 최신 Chrome에서 콘솔 에러 없이 작동해야 함.
* **법적 고지:** 개인정보처리방침(Privacy Policy) 및 이용약관(Terms of Service) 페이지 필수 포함.

### 4. Git 및 협업 프로세스 (Templates)
GitLab Merge Request(MR) 템플릿이 구체화되어, 체계적인 코드 리뷰와 히스토리 관리를 유도합니다.

* **삭제된 템플릿:** 단순한 형태의 `Add`, `Change`, `Update` 등.
* **추가/수정된 템플릿:** `Feat`, `Fix`, `Chore`, `Reword`, `Translation`.
* **템플릿 형식:** 단순 텍스트에서 아래와 같은 구조화된 양식으로 변경되었습니다.
    * 🔍 **What was changed:** 변경 내용 요약
    * 🎯 **Why it was changed:** 변경 이유
    * 🧩 **Impact:** 프로젝트 영향도 (Major/Minor)
    * 🧪 **How it was tested:** 테스트 방법
    * 👀 **Review expectation:** 원하는 리뷰 종류

### 5. 문서화 (README Requirements)
`README.md` 작성 요건이 매우 엄격해졌습니다. 평가의 중요한 척도가 됩니다.

* 팀원 및 역할(Role) 명시.
* 프로젝트 관리 방식 (협업 툴, 일정 관리 등).
* 기술 스택 선정 이유.
* 데이터베이스 스키마 다이어그램.
* 선택한 모듈 리스트와 점수 계산 내역.
* 각 멤버의 개별 기여도 상세 기술.

### 요약
이번 업데이트는 단순한 코딩 능력을 넘어 **"현업과 유사한 팀 프로젝트 경험"**을 제공하는 데 초점이 맞춰져 있습니다. 기획(PO), 관리(PM), 설계(Tech Lead)의 역할을 나누고, Git을 체계적으로 사용하며, 자유로운 주제 안에서 스스로 기술 스택(모듈)을 조합하여 목표 점수(14점)를 달성해야 합니다.

`ft_transcendence` 과제가 버전 4로 업데이트되면서 **평가 기준의 패러다임이 완전히 바뀌었습니다.** 단순한 기능 구현에서 **팀 프로젝트 운영 및 문서화**로 무게 중심이 이동했습니다.

요청하신 대로 **변경 사항 비교표**와 **기존 방식대로 수행했을 경우 탈락하게 되는 결정적인 호환 불가 요소**를 정리해 드립니다.

### 2. 기존 기준(V3) 수행 시 '호환되지 않는' 치명적 요소
만약 기존 과제(V3) 기준으로 프로젝트를 진행하다가 새로운 평가표(V4)로 평가를 받게 된다면, **기능이 완벽하게 작동하더라도 아래의 이유로 0점 처리되거나 과락할 가능성이 매우 높습니다.**

#### 🚨 1. 문서화 및 역할 부재 (즉시 감점/탈락 위험)
* **호환 불가:** 기존에는 코딩만 잘하면 됐지만, 이제는 **README.md**에 팀원별 **Role(PO, PM, Tech Lead)** 과 **구체적인 기여도**, **프로젝트 관리 방식(협업 툴 등)** 이 명시되어 있지 않으면 평가를 시작하자마자 감점되거나 `Rigor` 항목에서 Fail을 받게 됩니다.
* **평가 질문:** "각 팀원이 자신의 역할(Role)을 설명할 수 있는가?" (설명 못 하면 감점)

#### 🚨 2. 법적 필수 페이지 누락 (Project Rejection)
* **호환 불가:** 기존에는 게임만 잘 돌아가면 됐으나, V4에서는 웹사이트 내에 **Privacy Policy(개인정보처리방침)** 와 **Terms of Service(이용약관)** 페이지가 반드시 존재해야 합니다.
* **평가 기준:** "내용이 부실하거나 해당 페이지가 없으면 프로젝트는 **거부(Rejection)**됩니다."

#### 🚨 3. 점수 산정 방식의 괴리 (14점 미달)
* **호환 불가:** 기존의 '7개 메이저 모듈' 방식은 V4의 '14점 포인트 제도'와 1:1로 매칭되지 않을 수 있습니다.
    * 예: 기존에 'Standard User Management'를 구현했다면 V4에서도 점수를 받겠지만, 기존에 단순히 'Pong 게임' 자체에 집중했던 기능들은 V4에서 별도의 모듈 점수(Major 2점 등)로 인정받기 위해 **기술적 난이도에 대한 정당성(Justification)** 을 README에 써야 합니다. 이를 준비하지 않았다면 점수 미달로 탈락합니다.

#### 🚨 4. 기술 스택 및 아키텍처 불일치
* **호환 불가:** 기존에는 "Backend 없이" 구현하는 옵션이 있었으나, V4에서는 **Frontend + Backend + Database**의 3-Tier 구조가 강제됩니다. 로컬 스토리지에만 의존하는 앱을 만들었다면 탈락입니다.
* **호환 불가:** 기존 V3의 "순수 PHP 백엔드" 제약이 사라지고 프레임워크 사용이 권장됩니다. 만약 V3 기준에 맞춰 프레임워크 없이 날코딩(Hard-coding)을 했다면, V4의 **"Web - Use a Framework (Major/Minor)" 점수**를 획득하지 못해 14점을 채우기 어려워질 수 있습니다.
